Anatomy of an apex outage

How a leftover A record and an exact-match Route quietly sent my apex domain to a stranger's nginx — and what it taught me about DNS migrations, testing methods, and the order you remove things in.

The symptom

vawsome.com was intermittently serving a plain 404 Not Found — nginx/1.18.0 (Ubuntu) page. Not a Cloudflare error page. Not my Worker's own asset 404. A real nginx origin's default error page, from a server I don't run.

Meanwhile www.vawsome.com and vinay.vawsome.com — served by the exact same Cloudflare Worker — worked perfectly the whole time. Same code, same zone, three hostnames, one broken.

The architecture, briefly

The whole setup is one Cloudflare Worker fronting static assets, branching on the Host header: the apex and www get the company page, the vinay subdomain gets the personal site. For the Worker to answer, something has to bind each hostname to it. Cloudflare gives you two mechanisms that look interchangeable and very much aren't:

  • A Custom Domain creates and fully owns a special DNS record with no origin behind it. 100% of traffic for that host goes to the Worker, full stop.
  • A Route attaches to whatever DNS record already exists for a pattern. If the pattern doesn't match a request — or the record underneath still points somewhere real — traffic falls through.

The root cause

The zone had a leftover, proxied A record for the bare apex, pointing at a real nginx server from however the site was originally stood up. Layered on top was a Workers Route with the pattern vawsome.com — no wildcard — trying to intercept requests before they reached that origin.

The exact-match pattern silently failed to match real requests. So traffic sailed straight through Cloudflare's proxy to the live nginx origin, which had no vhost for the domain and answered with its default 404.

Why were the other two hostnames fine? www happened to match a separate wildcard Route (*.vawsome.com/*). And vinay.vawsome.com had been set up as a Custom Domain from the start — there was no origin behind it to fall through to. The one hostname configured the right way was the one that never broke. That's not a coincidence; that's the lesson.

The fix

  1. Add vawsome.com as a Custom Domain on the Worker.
  2. Confirm it serves the site correctly.
  3. Only then delete the apex A record and the broken Route.

I'm listing it in that order because I didn't do it in that order. Mid-fix, I deleted the broken Route before the Custom Domain was confirmed working — and caused a brief but total apex outage. No Route and no Custom Domain means nothing intercepting at all.

Sequence migrations as add-new → confirm-working → remove-old. Never remove-then-add. This is the same rule as blue-green deployments, and DNS is exactly where people forget it applies.

The debugging trap

The nastiest part wasn't the DNS — it was trusting my test methods. During the incident, curl and a real browser disagreed for a while, and there was genuine confusion about whether the nginx 404 was a testing artifact. A fresh browser navigation was what finally settled it.

When debugging "it works for me / it doesn't for you," verify with a real browser navigation, not curl alone — and don't trust a cached page either. Hard-refresh, or cache-bust with a throwaway query string.

What I'd tell you to remember

  • Prefer Custom Domains over Routes for binding hostnames to Cloudflare Workers. A Custom Domain owns the record; a Route negotiates with whatever's already there.
  • Leftover DNS records are live ammunition. The apex A record wasn't doing anything visible — until the layer above it failed and it quietly became the answer.
  • Partial failure patterns are a map. Three hostnames, one broken: the difference in how each was configured pointed straight at the cause.
  • Add, confirm, then remove. In that order. Always.

Sixteen years in, incidents like this one are still the best teachers — small enough to be harmless, structured exactly like the ones that aren't.
